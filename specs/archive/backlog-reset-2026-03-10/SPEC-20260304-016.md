### Task 2: Auth Foundation — Fix login.tsx

`packages/web/app/routes/login.tsx` has 2 TODOs:
```
// TODO: Call Convex auth:validatePassword mutation (line 23)
// TODO: Integrate Convex http actions with proper cookie-based session management (line 38)
```

**Minimal viable auth** (not full OAuth, just basic password-based):

1. Add `convex/auth.ts` with:
   ```typescript
   export const validatePassword = mutation({
     args: { username: v.string(), password: v.string() },
     handler: async (ctx, args) => {
       // Check against workspace settings (admin password from settings)
       const settings = await ctx.db.query("settings").first();
       const adminPassword = settings?.adminPassword;
       if (!adminPassword) return { success: true, token: "dev-mode" }; // dev mode: no password set = open
       const valid = args.password === adminPassword; // NOTE: Hash in production
       if (!valid) throw new Error("Invalid credentials");
       return { success: true, token: `agf_session_${Date.now()}` };
     }
   });

   export const setAdminPassword = mutation({
     args: { password: v.string() },
     handler: async (ctx, args) => {
       const existing = await ctx.db.query("settings").first();
       if (existing) await ctx.db.patch(existing._id, { adminPassword: args.password });
       else await ctx.db.insert("settings", { adminPassword: args.password });
     }
   });
   ```

2. Fix `login.tsx` to call `useMutation(api.auth.validatePassword)` and store the returned token in localStorage.

3. Add auth check to `DashboardLayout.tsx` — if no token in localStorage, redirect to /login.

4. Add "Set Password" field to Settings → General tab (calls `auth.setAdminPassword`).

NOTE: This is intentionally simple (no bcrypt, no JWT) — it's a self-hosted local tool. The goal is functional, not enterprise-grade.

### Task 3: Docker Sandbox Backend — Wire the UI Toggle

The Create Agent / Edit Agent dialog has a "Docker Sandbox" toggle (`sandboxEnabled: boolean`, `sandboxImage: string`). These are stored in the agents table. The backend never acts on them.

**What needs to happen when an agent with `sandboxEnabled: true` executes:**

In `convex/mastraIntegration.ts` (the `executeAgent` action), add sandbox awareness:
```typescript
// If agent.sandboxEnabled, use sandboxImage for execution context
if (agentConfig.sandboxEnabled && agentConfig.sandboxImage) {
  // Log that sandbox mode is active
  await ctx.runMutation(internal.logs.add, {
    agentId,
    level: 'info',
    message: `[Sandbox] Executing in Docker image: ${agentConfig.sandboxImage}`,
  });
  // NOTE: Actual Docker container spawning can't happen from Convex (serverless)
  // Instead: set an execution context flag and route to the CLI executor
  // For now: add a metadata field to the response indicating sandbox was requested
}
```

**CLI integration** — in `packages/cli/src/commands/chat.ts`:
When the agent response metadata includes `sandboxRequested: true`, show:
```
⚠  Agent has Docker Sandbox enabled (image: node:20-alpine)
   To execute with full sandbox isolation, run:
   agentforge sandbox run agent-001 --message "your message"
```

Add `agentforge sandbox run <agentId> --message <msg>` command that:
1. Checks if Docker is installed (`docker --version`)
2. Pulls the image if needed (`docker pull <image>`)
3. Runs the agent execution inside a container:
   ```bash
   docker run --rm -i <image> node -e "/* agent execution */"
   ```
4. Returns the result

This gives users a real, working sandbox even if it's CLI-only for now.
