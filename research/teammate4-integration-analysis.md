## Root Cause Analysis

**The placeholder is showing because the CLI's `dist/` directory was not rebuilt after AGE-91 landed.**

The chain of events:
1. Pre-AGE-91: `dist/default/dashboard/app/routes/chat.tsx` was built from an old version of the template (Feb 17 2026 timestamp). This old version had the `setTimeout + placeholder` pattern and no `convex/chat.ts`.
2. AGE-91 (commit `19aeb08`, merged Feb 18 2026): The template was updated correctly — `chat.tsx` was rewritten with real Convex action calls, and `convex/chat.ts` was added with real LLM execution via AI SDK.
3. **The CLI package was never rebuilt** — `pnpm build` was not run in `packages/cli/` after the merge. The `publicDir: 'templates'` in `tsup.config.ts` copies the templates directory to `dist/` during build — but only when the build runs.
4. The `dist/` directory is in `.gitignore` at the repo root, so it cannot be version-controlled. However it exists on disk (presumably from a previous build). When users scaffold a new AgentForge project using the CLI, they get whichever version of `dist/` is on disk at scaffold time.

**Secondary issue**: The `dist/default/convex/` directory has no `chat.ts` at all — only the old `messages.ts`, `threads.ts` etc. from before AGE-91. The new `convex/chat.ts` that the rewritten `chat.tsx` depends on does not exist in dist. This means even if a user scaffolds from the template source directly, the `api.chat.*` calls in `chat.tsx` would fail because `chat.ts` hasn't been scaffolded.

**The placeholder is a symptom of the dist being out of date, not a bug in the integration code itself.** The integration code (both template and cloud) is correct.

## Minimum Changes Needed

### Fix 1: Rebuild the CLI package
```bash
cd /Users/agent/Projects/Agentic-Engineering-Agency/agentforge/packages/cli

```
This runs `tsup` with `publicDir: 'templates'`, which copies the updated templates into `dist/`. This is the most direct fix. After rebuilding:
- `dist/default/dashboard/app/routes/chat.tsx` will have the real Convex action implementation
- `dist/default/convex/chat.ts` will exist with the full LLM pipeline

### Fix 2: Add a build check to CI
The root cause is that the dist was not rebuilt. CI should either:
- Run `pnpm build` in `packages/cli` before any integration tests/publish steps
- OR: not commit `dist/` to the repo at all (it's gitignored anyway) and always build on demand
- Add a CI step: `pnpm --filter @agentforge-ai/cli build`

### Fix 3: Convex environment variable setup
The `chat.ts` action requires at minimum ONE of:
- `OPENROUTER_API_KEY` (recommended — routes to all providers)
- `OPENAI_API_KEY`

These must be set as Convex environment variables (`npx convex env set OPENROUTER_API_KEY sk-or-...`). A missing key will cause the action to throw and the chat will display the error message, not a response.

The template's `.env.example` at `packages/cli/templates/default/.env.example` documents all supported env vars:
- `OPENAI_API_KEY` — OpenAI
- `OPENROUTER_API_KEY` — OpenRouter (recommended default)
- `ANTHROPIC_API_KEY` — Anthropic
- `GEMINI_API_KEY` — Google Gemini
- `XAI_API_KEY` — xAI/Grok

### Fix 4 (Optional): Update dist/chat.tsx API import path
The dist chat.tsx uses `import { api } from '@convex/_generated/api'` (path alias) while the template uses `import { api } from '../../convex/_generated/api'` (relative). Both should work with proper tsconfig path aliases in the scaffolded project — but this should be verified.

### Fix 5 (agentforge-cloud only): Add Chat UI to Dashboard
The `agentforge-cloud` dashboard (`apps/dashboard/src/routes/`) has **no chat route**. The backend is fully implemented:
- `convex/threads.ts` — `sendMessage` action (stores user msg + calls `runAgentForThread`)
- `convex/agentRunner.ts` — `runAgentForThread` action (runs `ConvexAgent.generate()`)
- `convex/mastra.ts` — `getModel()` for BYOK model resolution

A new chat UI route would need to call:
- `api.threads.createThread` — create a thread
- `api.threads.getMessages` — subscribe to messages
- `api.threads.sendMessage` — send message + trigger AI response
- `api.threads.getThreads` — list existing threads for an agent

**No backend changes are needed** for the agentforge-cloud chat — only a new frontend route/component.
