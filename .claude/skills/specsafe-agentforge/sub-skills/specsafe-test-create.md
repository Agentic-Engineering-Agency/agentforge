# SpecSafe Test Create

## Read First

1. active spec in `specs/active/`
2. `docs/SPECSAFE_WORKFLOW.md`
3. `prompts/test-designer.md`
4. `PROJECT_STATE.md`
5. `docs/SOURCE_OF_TRUTH.md` if external APIs affect test behavior

## Steps

1. Run `specsafe test-create <SPEC-ID>`.
2. Review the generated tests or test plan.
3. If upstream `specsafe test-create` reports `0 requirements`, confirm the spec uses the exact heading `### Functional Requirements` above the requirements table.
4. Add missing cases for approvals, audit logs, scheduling, realtime behavior, and security if relevant.
5. Keep the final spec contract tests under `tests/spec_*.test.ts` so GitHub Actions runs them through `pnpm test:specs`.
6. Remove tests that overreach beyond the active spec.
7. Run `pnpm spec:stage <SPEC-ID> TEST` if the spec has not yet been marked as `TEST`.
8. Re-run `pnpm spec:status` and check the spec header so the stage is consistent everywhere.

## Rules

- Prefer behavior and contract tests over implementation-detail tests.
- Keep the first test set small but representative.
- Flag unclear requirements instead of guessing.
- Do not implement production code in this stage.
- If autogenerated tests are low quality, replace them with smaller hand-curated tests tied to explicit spec requirements.
- Treat CI discoverability as part of test quality, not an optional follow-up task.
- Do not treat upstream `specsafe` stage reporting as authoritative after test creation; use `pnpm spec:status`.
- Treat upstream parser compatibility as a spec-format concern, not a reason to abandon the repo workflow.
